## ハッシュルーティングを採用する理由

ブラウザでルーティングを実装する方法は 2 種類:

| 方式 | URL 例 | サーバ設定 | Pros |
|---|---|---|---|
| ハッシュ | `/app/#/users` | **不要** | 静的ホスティング OK / 戻る進む自動 / 実装簡単 |
| History API | `/app/users` | **必要** (全パスを index.html に) | URL が綺麗 / SEO 対応しやすい |

本書 chibireact は **教育目的** なので、サーバ設定不要のハッシュ方式を採用します。
プロダクションでは History API ベースのほうが一般的ですが、本質は同じ:
- URL から「現在のパス」を取り出す
- パス変化を購読して再 render する
- リンククリックで URL を書き換える

## 実装: `useHashLocation`

ハッシュを購読する hook:

```ts
function getHashPath(): string {
  if (typeof window === 'undefined') return '/'
  const hash = window.location.hash
  if (!hash || hash === '#') return '/'
  return hash.startsWith('#') ? hash.slice(1) || '/' : hash
}

export function useHashLocation(): [string, (to: string) => void] {
  const [path, setPath] = useState(getHashPath())

  useEffect(() => {
    const listener = () => setPath(getHashPath())
    window.addEventListener('hashchange', listener)
    return () => window.removeEventListener('hashchange', listener)
  }, [])

  const navigate = (to: string) => {
    window.location.hash = to.startsWith('/') ? to : `/${to}`
  }

  return [path, navigate]
}
```

ポイント:
- `useState(getHashPath())` で初期パスを取得
- `useEffect` で `hashchange` イベントを購読
- cleanup で `removeEventListener` (Part 3.4 で習った副作用パターン)

## 実装: `Link`

`a` タグの薄いラッパ:

```tsx
export function Link(props: {
  to: string
  children?: ChibireactNode | ChibireactNode[]
}) {
  const handleClick = (e: MouseEvent) => {
    if (e.ctrlKey || e.metaKey || e.shiftKey || e.altKey) return
    // 修飾キーなしならブラウザのデフォルト挙動 (hash 書き換え) に任せる
  }
  const href = `#${props.to.startsWith('/') ? props.to : `/${props.to}`}`
  return createElement('a', { href, onClick: handleClick }, props.children)
}
```

ポイント:
- **`<a href="#/...">` を生成** するだけ
- ブラウザがデフォルトで hash を書き換えるので preventDefault 不要
- 修飾キー (Cmd / Ctrl) 押下時は通常クリック動作 (新タブ等) に任せる
- アクセシビリティ: 普通の `<a>` なのでスクリーンリーダや右クリックメニューが正常動作

## 実装: `Route` / `Switch`

```tsx
export function Route(props: {
  path: string
  component: () => ChibireactNode
}) {
  const [current] = useHashLocation()
  if (current !== props.path) return null
  return props.component()
}

export function Switch(props: { children?: ChibireactNode | ChibireactNode[] }) {
  const [current] = useHashLocation()
  const list = Array.isArray(props.children) ? props.children : [props.children]
  for (const child of list) {
    if (child.type === Route) {
      const path = child.props.path
      if (path === '*' || path === current) {
        return child.props.component()
      }
    }
  }
  return null
}
```

`Switch` は **最初にマッチした 1 つだけ** 描画。`'*'` は wildcard でフォールバック。

## 使用例

```tsx
const Home = () => createElement('h1', null, 'Welcome')
const Users = () => createElement('h1', null, 'Users')
const NotFound = () => createElement('h1', null, '404')

const App = () => createElement('div', null,
  createElement('nav', null,
    createElement(Link, { to: '/' }, 'Home'),
    ' | ',
    createElement(Link, { to: '/users' }, 'Users'),
  ),
  createElement(Switch, null,
    createElement(Route, { path: '/', component: Home }),
    createElement(Route, { path: '/users', component: Users }),
    createElement(Route, { path: '*', component: NotFound }),
  ),
)
```

## ちょっと深掘り

### History API ルーティングの実装

ハッシュ方式とほぼ同じです:

```ts
function getPath() {
  return window.location.pathname
}

export function useLocation() {
  const [path, setPath] = useState(getPath())

  useEffect(() => {
    const listener = () => setPath(getPath())
    window.addEventListener('popstate', listener)
    return () => window.removeEventListener('popstate', listener)
  }, [])

  const navigate = (to: string) => {
    window.history.pushState({}, '', to)
    setPath(to)  // popstate は pushState では発火しないので明示更新
  }
  return [path, navigate]
}
```

違い:
- `hashchange` → `popstate` (戻る進むのみ発火)
- `location.hash = ...` → `history.pushState({}, '', to)` + 手動 setState
- `<a href="/...">` → `<a href="/..." onClick={preventDefault + navigate}>` で fallback 必須

サーバが `/users` のような GET にも `index.html` を返すように設定する必要あり (NGINX try_files / Vite の historyApiFallback など)。

### パスパラメータの実装

```ts
function matchPath(template: string, actual: string): Record<string, string> | null {
  const t = template.split('/').filter(Boolean)
  const a = actual.split('/').filter(Boolean)
  if (t.length !== a.length) return null

  const params: Record<string, string> = {}
  for (let i = 0; i < t.length; i++) {
    if (t[i].startsWith(':')) {
      params[t[i].slice(1)] = a[i]
    } else if (t[i] !== a[i]) {
      return null
    }
  }
  return params
}

// '/users/:id' と '/users/42' をマッチング → { id: '42' }
```

これを `Route` に組み込めば `<Route path="/users/:id" component={UserDetail} />` が書けるようになります。
本書では割愛しますが 30 行程度で実装可能です。

### Suspense との組み合わせ

```tsx
<Switch>
  <Route path="/users" component={() => (
    <Suspense fallback={<Spinner />}>
      <UsersPage />
    </Suspense>
  )} />
</Switch>
```

ルート単位で Suspense を置くと、ページ切り替え時のローディング表示が自然に書けます。
本書 Part 5.4 の Suspense と組み合わせれば、データ取得込みのページ遷移が完成します。

### 「フレームワーク」としてのルーティング

Next.js / Remix のような **フルスタックフレームワーク** はルーティングを **ファイルベース** で表現します:

```
app/
├── page.tsx          → /
├── users/
│   ├── page.tsx      → /users
│   └── [id]/
│       └── page.tsx  → /users/:id
```

ファイル構造そのものがルート定義。手動で `<Route>` を書かなくて良い反面、ビルド時にルート情報を生成する必要があります。
本書スコープ外ですが、人気の方向性として知っておくと良いです。
