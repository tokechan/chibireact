## なぜ可視化が要るか

Part 1-5 でやってきたとおり、Fiber tree の構造は **概念上は分かる** ものの、**実際にどんな形になっているか** は脳内シミュレーションが大変です。

```tsx
const App = () => {
  const [count, setCount] = useState(0)
  return createElement(
    'div',
    null,
    count > 0 && createElement('p', null, `count: ${count}`),
    createElement(Counter, null),
  )
}
```

このコンポーネントの fiber tree は:

```
<App />
  div  [PLACEMENT]
    "0" or undefined
    <Counter />  [PLACEMENT]
      button  [PLACEMENT]
        "click"
```

…と、頭の中で組み立てる必要があります。`dumpFiberTree` があれば **実物を即座に見られる** ので、特に Part 2 (Fiber)・Part 3 (Hooks)・Part 5 (Suspense/ErrorBoundary) で挙動を確認しながら学べます。

## 実装: `dumpFiberTree`

```ts
function fiberToLine(fiber: Fiber): string {
  let label: string
  if (fiber.type === TEXT_ELEMENT) {
    label = `"${String(fiber.props.nodeValue ?? '')}"`
  } else if (typeof fiber.type === 'function') {
    label = `<${fiber.type.name || 'Anonymous'} />`
  } else if (typeof fiber.type === 'string') {
    label = fiber.type
  }

  const tags: string[] = []
  if (fiber.effectTag) tags.push(fiber.effectTag)
  if (fiber.suspended) tags.push('suspended')
  if (fiber.error !== undefined) tags.push(`error:${String(fiber.error)}`)
  if (fiber.key !== null && fiber.key !== undefined) {
    tags.push(`key=${JSON.stringify(fiber.key)}`)
  }

  return tags.length > 0 ? `${label}  [${tags.join(', ')}]` : label
}

export function dumpFiberTree(root: Fiber | null, indent = 0): string {
  if (root === null) return ''
  const lines: string[] = []
  walk(root, indent, lines)
  return lines.join('\n')
}

function walk(fiber: Fiber, depth: number, out: string[]): void {
  out.push('  '.repeat(depth) + fiberToLine(fiber))
  if (fiber.child) walk(fiber.child, depth + 1, out)
  if (fiber.sibling) walk(fiber.sibling, depth, out)
}
```

ポイント:
- 文字列・関数・HTML の 3 種類で **表記を切り替え**
- effectTag / suspended / error / key が立っていれば **タグ表示**
- 再帰で indent を増やす ((`'  '.repeat(depth)`)
- child は深さ +1、sibling は同じ深さ

## 実装: `findFiber`

```ts
export function findFiber(
  root: Fiber | null,
  predicate: (f: Fiber) => boolean,
): Fiber | null {
  if (root === null) return null
  if (predicate(root)) return root
  if (root.child) {
    const found = findFiber(root.child, predicate)
    if (found) return found
  }
  if (root.sibling) {
    const found = findFiber(root.sibling, predicate)
    if (found) return found
  }
  return null
}
```

DFS で最初にマッチする fiber を返す。

```ts
const counterFiber = findFiber(root, f => f.type === Counter)
console.log(counterFiber?.hooks)  // Counter の useState 状態を覗く
```

## 実装: `fiberStats`

```ts
export function fiberStats(root: Fiber | null) {
  let depth = 0, total = 0, hostElements = 0, functionComponents = 0, textNodes = 0

  function walk(f: Fiber, d: number) {
    depth = Math.max(depth, d)
    total++
    if (f.type === TEXT_ELEMENT) textNodes++
    else if (typeof f.type === 'function') functionComponents++
    else if (typeof f.type === 'string') hostElements++
    if (f.child) walk(f.child, d + 1)
    if (f.sibling) walk(f.sibling, d)
  }

  if (root) walk(root, 0)
  return { depth, total, hostElements, functionComponents, textNodes }
}
```

「このアプリの fiber tree は深さ 8 で 247 ノードある」みたいな全体像が掴める。

## React DevTools との比較

本物の [React DevTools](https://react.dev/learn/react-developer-tools) は:
- ブラウザ拡張で別パネルとして動く
- コンポーネントツリーを GUI で表示・展開
- **props / state / hooks** を選択時に表示
- **profiler** で render 時間を計測
- **どの hook がどのコンポか**追跡

これを実現するには:
- React 内部から hook を **「devtools 用の名前付き形式」** で公開
- 開発専用の `__REACT_DEVTOOLS_GLOBAL_HOOK__` を window に置く
- 拡張が定期的にツリーをスキャンして UI 更新

本書 chibireact では難しすぎるので CLI レベルで諦め、**`console.log(dumpFiberTree(root))`** で代替。

## ちょっと深掘り

### React DevTools が遅い問題

DevTools を開いていると React が **数倍遅く** なります。理由:
- DevTools が定期的にツリー全体をスキャン
- 各 fiber から hook 名・props を抽出
- 大規模アプリでは大量データを送受信

「Profiler を使うときだけ DevTools を開く」運用が現実的。本書 chibireact では存在しない問題。

### 自前 DevTools を作るなら

簡易版でも次が要ります:
1. `currentRoot` を window に公開する API
2. fiber の subscribe (commit ごとに更新)
3. UI: コンポーネント tree のレンダリング
4. ノードクリック → props/hooks 表示

UI 部分を chibireact 自身で書けば再帰的で面白い (chibireact が chibireact をデバッグ)。
**Web Worker** に処理を逃せば本物 React DevTools の遅延問題も避けられる。これは付録 (Part 7) で深掘り候補。

### `console.log` でも十分

実プロジェクトでも、**コンポ内に `console.log` を仕込む** だけで多くは分かります:

```tsx
function Counter() {
  const [count, setCount] = useState(0)
  console.log('Counter rendered', count)
  return ...
}
```

「何回 render したか / そのときの state」が即見える。
DevTools が無い環境 (RN, SSR) でも有効。**手軽に勝るデバッグツールはない**。

### React Compiler とデバッグ

React Compiler はコンパイル時に useMemo / useCallback を自動挿入するので、開発者が書いたコードと **実際に動くコードが一致しない** 問題が出ます。
DevTools 側で「コンパイル前の元コード」「コンパイル後のコード」を切り替えて見られるようにする UI が今後求められそう。本書スコープ外。
