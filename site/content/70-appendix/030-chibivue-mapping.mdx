# A.3 chibivue ↔ chibireact マッピング表

本書 chibireact は、ubugeeei 氏の [**chibivue**](https://book.chibivue.land/) (Vue を小さく作りながら学ぶ本) にインスパイアされ、その **React 版** を目指して書かれました。
本章は **chibivue を既に読んだ方** が「同じ概念は本書のどこ？」を素早く引けるための索引です。

両者の設計思想の **共通点と相違点** も並列で見ていきます。

## この章のゴール

- chibivue の各章 / Part に本書の対応箇所が即引ける
- Vue と React の **設計判断の違い** が章単位で見える
- 「同じ問題を別解で解いた」ペアを通じて両者の哲学が深まる

> 章末に動くもの: なし（マッピングテーブル）。

前章: [A.2 本家 React のソースコードを読む](./020-reading-react-source)

### Vue と React の最大の違いはここ (1.9)

**chibivue の小さなリアクティビティ**:
- Proxy ベース、`reactive()` / `ref()` で値の変更を追跡
- 値が変わると **「変わった部分だけ」が分かる** ので局所更新

**本書の useState**:
- 不変 state + 再 render (= 関数コンポを丸ごと再評価)
- どこが変わったかを知るには **Fiber tree の diff** が必要

両者は同じ問題 (UI の自動更新) を **真逆のアプローチ** で解決しています。

## 設計判断の対比

両書を読み比べたときに見えてくる **設計上の対称性**:

| 概念 | Vue / chibivue | React / chibireact |
|---|---|---|
| 状態管理 | Proxy + reactive ラベル | 不変 state + re-render |
| 変更検知 | きめ細かい (どこが変わったか分かる) | 全体再評価 (差分計算で部分更新) |
| テンプレート | HTML 風 + ディレクティブ | JSX (JS の式) |
| 副作用 | ライフサイクルフック / watch | useEffect / useLayoutEffect |
| 並行性 | (Vapor Mode で対応中) | Concurrent Rendering (lane) |
| コンポネント定義 | SFC (template/script/style) | 関数 (JSX を return) |
| 親 → 子 | props | props |
| 子 → 親 | emit | コールバック props |
| 子 → 子孫 | provide / inject | useContext |

**「同じ問題を別解で解く」** のが両書を読み比べる最大の楽しみ。

## ちょっと深掘り

### 「両方理解する」ことの価値

フロントエンドエンジニアが Vue と React の両方を学ぶと:
- **デザインパターンの引き出し** が倍になる
- 自分が書いてるフレームワークの「**当たり前じゃない選択**」が見える
- ライブラリ選択時の判断軸が明確に
- 「reactive な思考」と「再評価な思考」を意識的に使い分けられる

「フレームワークに縛られない」エンジニアになる入り口として、本書 + chibivue の組み合わせは強力です。

### Solid.js / Svelte / Qwik も加えると

Solid.js は **Proxy ベースだが JSX を書く** という変わり種。
Svelte は **コンパイル時に reactive コードを生成** する別解。
Qwik は **resumability** (実行を中断・再開) という新コンセプト。

各フレームワークが「同じ問題」をどう解いているかの比較は、**フロントエンドの哲学** を学ぶ最良の方法です。

### chibivue 著者 ubugeeei さんへの謝辞

本書 chibireact は ubugeeei 氏の chibivue にインスパイアされて書かれました。
- 「フレームワークを小さく作りながら学ぶ」コンセプト
- 章を細かく刻む構成
- TDD ベースで進める方針 (本書独自要素)

すべて chibivue がベースとなっています。先行者へのリスペクトとともに、両書を読み比べることをおすすめします。
