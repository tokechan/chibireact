# 再レンダリングのトリガーと差分適用

[1.9 useState](./090-use-state) で「setState すると再レンダ」が動くようになりました。でも、こんなパターンに問題があります:

```tsx
const Counter = () => {
  const [a, setA] = useState(0)
  const [b, setB] = useState(0)

  return createElement('button', {
    onClick: () => {
      setA(a + 1)  // 再レンダ #1
      setB(b + 1)  // 再レンダ #2
    },
  }, `${a}-${b}`)
}
```

クリック 1 回で **2 回も再レンダが発生** します。3 つ呼べば 3 回。これは無駄です。
React 18 の **automatic batching** に倣い、**同じ tick の複数 setState を 1 回の再レンダにまとめます**。

> **本章のスコープについて**: 章タイトルに「差分適用」とありますが、本格的な diff は Part 2 (Fiber) で実装します。本章はトリガーの最適化（バッチング）に集中し、差分計算自体は「container.innerHTML = '' して全描画」のままです。これは意図的な分割で、Fiber 抜きで diff を書くと教育的に複雑すぎるためです。

## この章のゴール

- 同じ tick で setState を複数回呼んでも **再レンダは 1 回** に
- 別々のタイミング（次クリック等）の setState は別々に再レンダ
- 「最終状態」だけが描画される（途中の中間状態は描画されない）

> 章末に動くもの: 3 個のバッチングテストが緑になり、累計 64/64 緑。`setA(); setB(); setC()` が 1 再レンダ。

前章: [useState の最小実装](./090-use-state)

## 前提知識

- JavaScript のイベントループとマイクロタスク
- `queueMicrotask()` の挙動

## マイクロタスクキューを使う

ブラウザ JS には 2 つのキューがあります:

| キュー | 例 | タイミング |
|---|---|---|
| **マクロタスク** | `setTimeout`, `requestAnimationFrame` | 各 tick の合間 |
| **マイクロタスク** | `Promise.then`, `queueMicrotask` | 現在の同期コードが終わった**直後** |

setState を呼んだとき、**マイクロタスクとして再レンダをスケジュールする** と、同じイベントハンドラ内の複数 setState はすべて「現在の同期コードの終わり」を待ちます。すべて溜まったあとで 1 回だけ再レンダが走ります。

```
[click イベント開始]
  setA(1)    ← rerender をマイクロタスクキューに登録（既に登録済みならスキップ）
  setB(2)    ← 既にスケジュール済みなのでスキップ
  setC(3)    ← 同上
[同期コード終了]
[マイクロタスク実行]
  rerender() ← 1 回だけ
```

これがバッチングの基本原理です。

## 設計を考える

`hooks-state.ts` に「スケジュールされてるか」のフラグと、`queueMicrotask` を追加します。

```ts
let _scheduled = false

function _scheduleRerender(): void {
  if (_scheduled) return  // 既に登録済みなら何もしない
  _scheduled = true
  queueMicrotask(() => {
    _scheduled = false
    _rerender?.()
  })
}
```

そして setState の中で `_rerender?.()` を呼んでいたところを `_scheduleRerender()` に置き換えます。

## テストを書く（Red）

`packages/@chibireact/core/tests/hooks-batching.test.ts`:

```ts
import { describe, it, expect, beforeEach } from 'vitest'
import { createRoot } from '../src/create-root'
import { createElement } from '../src/create-element'
import { useState, _clearHooksForTesting } from '../src/hooks-state'

const flushMicrotasks = () => new Promise<void>((r) => queueMicrotask(r))

describe('useState batching (Part 1.10)', () => {
  let container: HTMLElement
  beforeEach(() => {
    container = document.createElement('div')
    _clearHooksForTesting()
  })

  it('同じ tick の複数 setState は 1 回の再レンダにまとめられます', async () => {
    let renderCount = 0
    const App = () => {
      renderCount++
      const [c, setC] = useState(0)
      return createElement('button', {
        onClick: () => {
          setC(prev => prev + 1)
          setC(prev => prev + 1)
          setC(prev => prev + 1)
        },
      }, c)
    }

    const root = createRoot(container)
    root.render(createElement(App, null))
    const initial = renderCount

    ;(container.firstChild as HTMLButtonElement).click()
    await flushMicrotasks()

    expect(renderCount).toBe(initial + 1) // 3 回 setState → 再レンダは 1 回
    expect(container.firstChild!.textContent).toBe('3') // 状態は 3 つ全部反映
  })
})
```

（実際の test には他に「別クリックは別再レンダ」「複数 useState の同時更新」も含めて 3 ケース）

実行すると、現状は 1 setState ごとに即再レンダしているので、`renderCount` が `initial + 3` になり失敗します。

## 実装する（Green）

`packages/@chibireact/core/src/hooks-state.ts` に追加:

```ts
let _scheduled = false

function _scheduleRerender(): void {
  if (_scheduled) return
  _scheduled = true
  queueMicrotask(() => {
    _scheduled = false
    _rerender?.()
  })
}

// setState 内で
const setState: Dispatch<T> = (action) => {
  // ... 値計算
  if (Object.is(hook.state, next)) return
  hook.state = next
  _scheduleRerender()  // ← 直接 _rerender() を呼ばずスケジュール
}
```

これだけ。テスト再実行:

```
 ✓ tests/hooks-batching.test.ts (3 tests)

 Test Files  9 passed (9)
      Tests  64 passed (64)
```

これで **Green**。

## 既存テストの async 化

`hooks-state.test.ts` の旧テストは「クリック後すぐに DOM が更新される」前提でした。バッチング導入で **マイクロタスク待ちが必要** になったので、各テストを `async` にして `await flushMicrotasks()` を入れます。

これは破壊的変更ではなく、より React 本家の挙動に近づいたことを意味します。React の `act()` ヘルパーや `@testing-library/react` の `userEvent` も内部で同様にマイクロタスクを待っています。

## 試してみる

ブラウザで動かすと、`onClick` で 3 回 setState しても **画面のチカチカは 1 回** だけです。React Devtools が手元になくても、`renderCount` を画面に出して確認できます:

```tsx
let renders = 0
const App = () => {
  renders++
  const [c, setC] = useState(0)
  return createElement('div', null,
    createElement('p', null, `count = ${c}`),
    createElement('p', null, `renders = ${renders}`),
    createElement('button', {
      onClick: () => {
        setC(c + 1)
        setC(c + 2)
        setC(c + 3)
      },
    }, '3 つ setState'),
  )
}
```

ボタンを 1 回クリックすると `renders` が 1 だけ増えます（`c` は 3 になる）。

## ちょっと深掘り

### `setTimeout(0)` ではなぜ駄目か

`queueMicrotask` の代わりに `setTimeout(fn, 0)` でも似たことができそうですが、**マイクロタスクの方が早い** ので推奨です:

```
setState → マクロタスクキュー (setTimeout)
[click イベント終了]
[他のマクロタスクが先に走る可能性]
[ようやく rerender]
```

マイクロタスクは「**同期コード終了の直後**」に必ず走るので、ユーザーから見て「clickの応答」と「再レンダ」がほぼ同時に見えます。

### React 18 の automatic batching との違い

React 18 の正式な automatic batching は、より精緻です:
- **イベントハンドラ内**だけでなく、Promise・setTimeout・ネイティブイベントまで一律でバッチ
- React 専用の **scheduler** が priority 付きで管理
- 緊急度の高い更新は `flushSync()` で同期的に走らせられる

本書 chibireact の実装は **マイクロタスクで一律バッチ** という最小版です。ほとんどのケースは正しく動きますが、`setTimeout` 内での setState などはこの最小版でも 1 回にまとまります（マイクロタスクキューが共通なので）。

### diff はなぜ Part 2 で扱うのか

「同じ tag なら DOM を再利用、違う tag なら作り直し」というナイーブな diff も書けます。しかし:

- 子要素の **位置ずれ** に対応するには key が必要
- key 付きの diff は「list reconciliation algorithm」と呼ばれ、複雑
- 本格的に書くと Fiber と同じくらいの複雑度になる

そのため本書では **Fiber と一緒に diff を実装する** 方針を採ります。Part 1.10 の段階では「再レンダ自体は全描画だが、トリガーは賢い」状態です。

### 入力フォーカスはまだ消える

`<input />` を含む UI で typing すると、現状はやはり **1 文字ごとに focus が消えます**:

```tsx
const App = () => {
  const [text, setText] = useState('')
  return createElement('input', {
    value: text,
    onInput: (e) => setText(e.target.value),
  })
}
```

これは バッチングの問題ではなく **「container.innerHTML = '' してるから input が再生成される」** 問題です。Fiber + diff で本格的に解消します（Part 2 のスコープ）。

## まとめ

- `queueMicrotask` で再レンダをスケジュール、フラグでまとめる
- 同じ tick の複数 setState は 1 再レンダ
- 別 tick の setState は別再レンダ
- 既存テストは async 化が必要
- diff 自体（賢い更新）は Part 2 で本格化
