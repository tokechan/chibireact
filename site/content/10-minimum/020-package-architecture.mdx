# パッケージ設計

[1.1](./010-create-root-api) で `createRoot` を実装したとき、コードは `packages/@chibireact/core/` に置きました。**なぜそんな深い場所に？** 普通の Node プロジェクトなら `src/` 直下で十分そうなのに。

この章では、モノレポ構成を採用した理由と、pnpm workspaces の仕組みをじっくり見ていきます。**手を動かす章ではなく「理解する章」**です。

## この章のゴール

- chibireact が **モノレポ** であることと、その理由を説明できる
- `pnpm-workspace.yaml` の役割が分かる
- スコープ付きパッケージ名 (`@chibireact/core`) の意味が分かる
- `pnpm --filter` で特定パッケージにコマンドを向けられる
- 将来追加するパッケージ（reconciler / jsx-runtime / server）の見通しが立つ

> 章末に動くもの: `pnpm --filter @chibireact/core test` でテストが実行され、その仕組みを説明できる状態。

前章: [初めてのレンダリングと createRoot API](./010-create-root-api)

## なぜモノレポか

「`@chibireact/core` を別の repo にする選択肢もあったのでは？」と思うかもしれません。実際、それは可能です。でも本書ではモノレポを採用しました。理由は 4 つ:

### 理由 1. **同じ commit でコード本文と書籍本文が同期**

例えば「createElement の API を変えました」という変更があったとき、**コード変更と章の変更を 1 つの PR でレビュー**できます。別 repo だと「コード PR と書籍 PR で齟齬が出ていないか」を別途確認する手間が生じます。

### 理由 2. **写経する読者が clone しやすい**

```bash
git clone git@github.com:tokechan/chibireact.git
cd chibireact
pnpm install   # site と core を一括セットアップ
pnpm dev       # site が起動
pnpm test      # core のテストが走る
```

1 つの clone で「読みながらコードを動かす」体験が完結します。

### 理由 3. **将来の `@chibireact/reconciler` 追加が楽**

Part 2 で Fiber 実装を別パッケージに切り出します。モノレポなら `packages/@chibireact/reconciler/` を作って `pnpm install` するだけで、既存の `core` から相対パスでインポートできます。

### 理由 4. **同じバージョンの React / TypeScript を共有**

`React 19` を `site/` でも `core` でも使う場合、モノレポなら `pnpm-lock.yaml` 1 つで版を統一できます。別 repo だと両方を手動で同期する手間が出ます。

> **逆に、別 repo にしないデメリット**: chibireact の `core` を独立して NPM 公開したくなったとき。本書では公開を想定していないので問題ないですが、将来公開するならモノレポから個別パッケージに「export」する仕組み（Lerna, Nx, Turborepo など）を追加することになります。

## pnpm workspaces の仕組み

ルートに 1 行の宣言ファイルがあります。

`pnpm-workspace.yaml`:

```yaml
packages:
  - 'site'
  - 'packages/@chibireact/*'
```

これで pnpm に「ルートからこのパスにある `package.json` を、すべて 1 つの workspace の構成要素として扱って」と伝えています。`packages/@chibireact/*` のように **glob パターン** が使えるので、新しいパッケージを `packages/@chibireact/reconciler/` のように作るだけで自動認識されます。

ルートの `package.json` も大事です:

```json
{
  "name": "chibireact",
  "version": "0.0.1",
  "private": true,
  "scripts": {
    "dev": "pnpm --filter chibireact-site dev",
    "test": "pnpm --filter '@chibireact/*' test"
  }
}
```

ポイント:
- `"private": true` — workspace ルートは NPM に公開しないので必須
- `scripts` の `pnpm --filter ...` で **特定の workspace package にコマンドを送る**
- ルートで `pnpm dev` と打つと `site/` の `dev` が動く
- ルートで `pnpm test` と打つと `@chibireact/*` 全部の `test` が走る

## `pnpm --filter` を試してみる

ターミナルで実際に動かしてみましょう。

```bash
# core パッケージだけのテスト実行
pnpm --filter @chibireact/core test

# core パッケージだけ依存追加
pnpm --filter @chibireact/core add -D some-dep

# site パッケージだけ dev 起動
pnpm --filter chibireact-site dev

# 全 workspace で test
pnpm -r test
```

`-r` は `--recursive` の短縮形で、全 workspace に対して同じコマンドを走らせます。`--filter` は対象を絞ります。

## なぜ `@chibireact/core` というスコープ付き名前か

NPM のパッケージ名には 2 種類あります:

| 名前の形 | 例 | 用途 |
|---|---|---|
| **スコープなし** | `react`, `lodash` | グローバル名前空間（早い者勝ち）|
| **スコープ付き** | `@types/node`, `@chibireact/core` | `@組織名/パッケージ名` 形式 |

スコープ付きのメリット:

1. **名前衝突しない** — `@chibireact/core` と `@anotherproject/core` は別物として扱える
2. **関連パッケージをグループ化できる** — `@chibireact/core`, `@chibireact/reconciler`, `@chibireact/jsx-runtime` で「同じプロジェクトの一部」と一目瞭然
3. **NPM 公開時に組織アカウントで管理できる** — 個人で `@your-name/...` も可
4. **本家 React への敬意** — 本家は `react` という単独名なのに対し、こちらは「自作の R 実装」なので別空間に置く

> **注**: `packages/@chibireact/` のディレクトリ構造（`@` を含む階層名）は pnpm では特別な扱いではありません。単なるディレクトリ名です。`package.json` の `"name": "@chibireact/core"` 側がスコープを表現しています。

## ちょっと深掘り

### 「workspace の中で互いを参照できるの？」

できます。例えば `@chibireact/jsx-runtime` から `@chibireact/core` を使いたいとき:

```json
// packages/@chibireact/jsx-runtime/package.json
{
  "name": "@chibireact/jsx-runtime",
  "dependencies": {
    "@chibireact/core": "workspace:*"
  }
}
```

`workspace:*` は「この workspace の中の同名パッケージ」を意味する pnpm 独自記法です。`pnpm install` 時にローカルの `packages/@chibireact/core/` がシンボリックリンクで参照されます。

### `node_modules` がルートに集中する仕組み

pnpm は workspace 内の依存を **ルートの `node_modules`** に集約します。`site/node_modules` や `packages/@chibireact/core/node_modules` ができても、その中身は基本的にルートへのシンボリックリンクで、**実体のディスク使用量は 1 つ分**です。これが pnpm の **content-addressable store** の効果。

### `.gitignore` の重要性

ルート `.gitignore` で `node_modules/` を除外していないと、workspace install 直後に **数万ファイルがコミット候補**になります（ラウンド 2 のリファクタで一度ヒヤッとしました）。
モノレポでは特に大事なので、`/node_modules/` ではなく `node_modules/` のように **ルート以外のすべてに効く書き方** を推奨します。
