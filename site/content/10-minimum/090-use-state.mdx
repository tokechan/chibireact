# useState の最小実装

ついにここまで来ました。**状態を持つ関数コンポーネント**を chibireact で実現します。これができれば、こんなコードが動きます:

```tsx
function Counter() {
  const [count, setCount] = useState(0)
  return createElement('button',
    { onClick: () => setCount(count + 1) },
    count,
  )
}

createRoot(container).render(createElement(Counter, null))
```

クリックするたびに数字が増えるボタン。**普通の React アプリ** です。

## この章のゴール

- `useState(initial)` で状態を持てる
- `setState(next)` で再レンダリングがトリガーされる
- `setState(prev => next)` の関数型更新も動く
- 同じ値を setState したら**再レンダしない**（`Object.is` 比較）
- 複数の `useState` を同じコンポーネント内で使える
- オブジェクトや配列など複合型も状態にできる

> 章末に動くもの: 5 個の useState テストが緑になり、累計 **61/61** 緑。Counter / Form / Object 状態が実際に動く。

前章: [関数コンポーネントの最小実装](./080-function-components)

## 前提知識

- React の `useState` の使い方（読者として）
- クロージャと「外部変数を関数の中から参照する」パターン
- `Object.is()` の挙動（`===` との違いは NaN と +0/-0 のみ）

## なぜ普通のローカル変数では駄目なのか

「コンポーネント内で `let count = 0` すれば？」と思うかもしれません。実は駄目です:

```tsx
function Counter() {
  let count = 0  // 毎回 0 にリセットされる
  return createElement('button',
    { onClick: () => { count++ } },  // 増えるが…
    count,
  )
}
```

問題:
1. **コンポーネントが呼ばれるたびに `count = 0` で初期化される**
2. `count++` してもコンポーネントが再評価されない（再レンダがトリガーされない）
3. たとえ再レンダしても、また 0 から始まる

`useState` が解決すること:
1. **値を関数の外に保管する**（再呼び出しでも残る）
2. setState で **再レンダをトリガー** する
3. 再レンダでも前回の値を返す

## 設計を考える: 「外」とはどこか

useState の「外」は、技術的にはモジュールスコープ or root スコープです。本書 chibireact では **モジュールスコープに hooks 配列**を持つ最小実装を採ります。

```
[render 開始]
  index = 0 にリセット
  Counter() を呼ぶ
    useState(0) → hooks[0] が無いので作る → return [0, setState]
    index = 1
  描画
[クリック: setState(1)]
  hooks[0].state = 1
  rerender() を呼ぶ
[render 開始 (2 回目)]
  index = 0 にリセット
  Counter() を呼ぶ
    useState(0) → hooks[0] がある → return [1, setState]  ← 前回の値!
  描画
```

「**hooks 配列の同じインデックスを毎回読みに行く**」のがポイント。これが React の **「Hooks のルール: 条件分岐の中で呼ばない」** が必要な理由です。順序が変わると別のインデックスを読んでしまうため。

## 実装する

### Step 1: hooks 状態モジュール

`packages/@chibireact/core/src/hooks-state.ts`:

```ts
type Hook<T = unknown> = { state: T }

let _rootHooks: Hook[] = []
let _index = 0
let _rerender: (() => void) | null = null

export type SetStateAction<T> = T | ((prev: T) => T)
export type Dispatch<T> = (action: SetStateAction<T>) => void

export function useState<T>(initial: T): [T, Dispatch<T>] {
  const index = _index++

  if (_rootHooks[index] === undefined) {
    _rootHooks[index] = { state: initial }
  }

  const hook = _rootHooks[index] as Hook<T>

  const setState: Dispatch<T> = (action) => {
    const next = typeof action === 'function'
      ? (action as (prev: T) => T)(hook.state)
      : action
    if (Object.is(hook.state, next)) return // 同じ値ならスキップ
    hook.state = next
    _rerender?.()
  }

  return [hook.state, setState]
}

/** @internal */
export function _resetHooks(rerender: () => void): void {
  _index = 0
  _rerender = rerender
}
```

ポイント:
- `_index++` で「呼ばれた順番」をインデックスにする
- 初回は箱を作る、2 回目以降は既存の値を返す
- setState のクロージャが `hook` オブジェクトを掴み続ける
- `Object.is` で同値判定 → 無駄な再レンダを防ぐ

### Step 2: createRoot を再レンダ対応にする

`packages/@chibireact/core/src/create-root.ts`:

```ts
import type { ChibireactNode } from './types'
import { render as renderToContainer } from './render'
import { _resetHooks } from './hooks-state'

export type Root = {
  render(element: ChibireactNode): void
}

export function createRoot(container: HTMLElement): Root {
  let lastElement: ChibireactNode | null = null

  const rerender = (): void => {
    if (lastElement === null) return
    container.innerHTML = ''
    _resetHooks(rerender)
    renderToContainer(lastElement, container)
  }

  return {
    render(element: ChibireactNode): void {
      lastElement = element
      container.innerHTML = ''
      _resetHooks(rerender)
      renderToContainer(element, container)
    },
  }
}
```

`render` 時に最後の element を覚えておき、setState から `rerender()` が呼ばれたら同じ element をもう一度描画する仕組み。

import { createRoot, createElement, useState } from '@chibireact/core'

function Counter() {
  const [count, setCount] = useState(0)

  return createElement('div', null,

## ちょっと深掘り

### Hooks のルール「条件分岐の中で呼ばない」の理由

React 公式が言う「Hooks のルール」は次の 2 つ:
1. ループ・条件・ネスト関数の中で呼ばない
2. React 関数（コンポーネント / カスタムフック）の中でだけ呼ぶ

このうち 1 は **本書の実装で破ると即死**します:

```tsx
function App() {
  if (someCond) {
    const [a] = useState('a')  // hooks[0] = { state: 'a' }
  }
  const [b] = useState('b')    // someCond=true なら hooks[1]、false なら hooks[0]
}
```

`someCond` の真偽でインデックスがずれて、`b` が `'a'` の値を読んでしまうバグになります。本家 React も同じ理由でこのルールを要求します（Fiber でも hook list は順序依存）。

### 関数型 setState (`setCount(c => c + 1)`) のメリット

```tsx
// Pattern A: 値を直接渡す
setCount(count + 1)

// Pattern B: 関数を渡す
setCount(prev => prev + 1)
```

クリックを連続したとき、A だとクロージャが古い `count` を掴んでいるため正しくインクリメントされません（古い値 + 1 を 3 回設定 = 1 回しか進まない）。

B は **「最新の値」が引数で渡される** ので、必ず正しくインクリメントされます。本書 chibireact もこの「関数渡しなら最新値を渡す」仕様を実装しています。

### `Object.is` を使う理由（`===` ではなく）

ほぼ `===` と同じですが、2 箇所違います:
- `Object.is(NaN, NaN)` → `true` （`===` は false）
- `Object.is(+0, -0)` → `false` （`===` は true）

実用的には NaN の扱いだけが効きます。`setState(NaN)` を 2 回呼んだら 2 回目をスキップしたい、という意図に React と本書ともに合っています。

- `useState(initial)` をモジュール状態 + index 配列の最小実装で書きました
- setState で `container.innerHTML = ''` → 全描画 という最も愚直な再レンダ
- Counter / Form / Object state まで動く（テスト 5 個 緑）
